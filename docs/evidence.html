Claims and evidence notes

This page records which repository artifacts support the major public-facing claims on the project page. Each item is phrased conservatively on purpose.

The repository is organized around two downstream evaluation tracks

  • README.md lines 3-8 define Track 1 as point cloud → range image → point cloud with PointPillars, and Track 2 as point cloud → range image with RangeDet.
  • README.md lines 64-98 link those tracks to their current execution scripts.

The active method is a 5-channel range-image codec with ROI-aware importance control

  • research_progress.md lines 16-38 summarize the current architecture: 5-channel input, resnet/darknet backbone, pp_lite head, adaptive quantization, decoder, and optional teacher distillation.
  • src/main_train.py lines 31-219 expose the training flags for quantizer mode, ROI target mode, distillation, head variants, decoder type, and detector-aware auxiliaries.
  • src/models/compression.py lines 166-200 show the adaptive quantizer and ROI importance head being wired into the top-level model.

Track 2 now has a credible baseline, so the codec is the main bottleneck

  • research_journal_en.md lines 221-248 say explicitly that the repaired raw/basic RangeDet baseline answered the “does the detector work?” question.
  • results_summary_current.md lines 41-61 report AP3D@0.5 = 0.4979 for raw/basic, versus near-zero values for reconstructed settings.

Track 1 reconstruction is still the bottleneck even when the detector side looks stable

  • results_summary_current.md lines 7-18 keep the identity-domain PointPillars runs around ~73 mAP3D(mod).
  • results_summary_current.md lines 20-37 show reconstructed endpoint results far below their paired references, with the best current entry still at 2.192242.
  • paper_fair_comparison_table.md lines 123-131 preserve one explicit detector-endpoint pair where the reference and reconstructed results are dramatically separated.

Oracle ROI results suggest there is still quantizer upside

  • research_progress.md lines 183-206 interpret native-vs-oracle results as evidence that importance-map quality remains the immediate bottleneck.
  • paper_fair_comparison_table.md lines 65-79 quantify the approximate entropy reduction under oracle ROI forcing.

Projection-path sparsification is a serious pre-codec bottleneck

  • research_progress.md lines 279-307 report raw vs identity vs reconstructed decomposition, including mean retention of 0.4169.
  • research_progress.md lines 311-326 keep that sparsification issue in the current-status section rather than burying it in old notes.

The immediate experimental direction is Track 2 root-cause recovery, not broad storytelling

  • research_journal_en.md lines 250-272 enumerate pilot branches A-D and the dependency order across the submitted jobs.
  • research_journal_en.md lines 234-247 keep the focus on valid-mask corruption, banding, and detector-agnostic objectives as the leading hypotheses.
